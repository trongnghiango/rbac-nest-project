Chào bạn, dựa trên 3 hình ảnh bạn cung cấp, hệ thống nhân sự này được tổ chức rất bài bản và chi tiết theo dạng **Ma trận (Matrix)** kết hợp **Phân cấp (Hierarchical)**. 

Có 4 yếu tố cốt lõi:
1. **Sơ đồ tổ chức (Org Chart):** Cây phân cấp từ Công ty -> Chi nhánh -> Phòng ban -> Nhóm.
2. **Cấp bậc (Grades/Levels):** Từ Bậc 1 đến Bậc 10 (CEO, Quản lý C1, Chuyên viên B2, Trợ lý A1...). Đây là thang đo chuẩn của toàn công ty.
3. **Chức danh thực tế (Job Titles):** Tùy vào mỗi phòng ban mà Cấp bậc sẽ có Tên chức danh khác nhau (VD: Cùng là Bậc 6, nhưng ở P.HCKT là "CV KTTC-HCNS", ở P.Dịch vụ lại gọi chung là "Chuyên viên B2").
4. **Vị trí làm việc (Locations):** HCM, HN...

Để xây dựng một hệ thống chuẩn chung (Core System) đáp ứng được mô hình này và có thể mở rộng (scale) cho các mô hình công ty khác trong tương lai, chúng ta cần thiết kế Database theo hướng **Data-Driven (Động)**, tuyệt đối không hard-code.

Dưới đây là thiết kế Kiến trúc và Database (sử dụng Drizzle ORM khớp với project NestJS của bạn).

### 1. Thiết Kế Cơ Sở Dữ Liệu (Database Schema)

Chúng ta sẽ bóc tách hệ thống thành các bảng (tables) độc lập và dùng một bảng trung gian `user_assignments` để nối chúng lại.

#### A. Các bảng Danh mục chuẩn (Master Data)

```typescript
import { pgTable, serial, text, varchar, integer, boolean, timestamp, foreignKey, uniqueIndex } from 'drizzle-orm/pg-core';
import { relations } from 'drizzle-orm';
import { users } from './users.schema'; // Bảng users hiện tại của bạn

// 1. Bảng Vị trí địa lý (Locations) - HCM, HN, DN...
export const locations = pgTable('locations', {
  id: serial('id').primaryKey(),
  code: varchar('code', { length: 50 }).unique().notNull(),
  name: varchar('name', { length: 255 }).notNull(),
  isActive: boolean('is_active').default(true),
});

// 2. Bảng Cấp Bậc (Grades/Levels) - Theo Image 3
export const grades = pgTable('grades', {
  id: serial('id').primaryKey(),
  levelNumber: integer('level_number').notNull(), // 1, 2, 3 ... 10 (Bậc)
  code: varchar('code', { length: 50 }).unique().notNull(), // B2, A1, C1...
  name: varchar('name', { length: 255 }).notNull(), // Chuyên viên B2, Trợ lý A1...
});

// 3. Bảng Đơn vị tổ chức (Organization Units) - Áp dụng mô hình Adjacency List (Tự tham chiếu)
// Xử lý Image 2 (Sơ đồ tổ chức phân nhánh)
export const orgUnits = pgTable('org_units', {
  id: serial('id').primaryKey(),
  parentId: integer('parent_id'), // Trỏ về chính bảng org_units. Null = HĐTV (Gốc)
  type: varchar('type', { length: 50 }).notNull(), // COMPANY, BRANCH, DEPARTMENT, TEAM
  code: varchar('code', { length: 50 }).unique().notNull(),
  name: varchar('name', { length: 255 }).notNull(), // P.Dịch vụ, Nhóm 1...
});

export const orgUnitsRelations = relations(orgUnits, ({ one, many }) => ({
  parent: one(orgUnits, { fields: [orgUnits.parentId], references: [orgUnits.id] }),
  children: many(orgUnits), // Lấy danh sách phòng ban con
}));

// 4. Bảng Chức danh/Chức vụ (Job Titles)
// Ánh xạ cụ thể Tên gọi của một chức vụ
export const jobTitles = pgTable('job_titles', {
  id: serial('id').primaryKey(),
  name: varchar('name', { length: 255 }).notNull(), // Giám đốc, Trưởng phòng, CV KT-THUẾ...
});
```

#### B. Bảng Cốt lõi: Lịch sử công tác (User Assignments)
Đây là bảng quan trọng nhất để giải quyết Image 1 (Danh sách NV). Một nhân viên có thể luân chuyển nhiều phòng ban, nên ta dùng bảng Assignment để lưu lại lịch sử làm việc hiện tại và quá khứ.

```typescript
// 5. Bảng Phân công Nhân sự (User Assignments)
export const userAssignments = pgTable('user_assignments', {
  id: serial('id').primaryKey(),
  userId: integer('user_id').references(() => users.id).notNull(),
  
  // Liên kết 4 chiều:
  orgUnitId: integer('org_unit_id').references(() => orgUnits.id).notNull(), // Làm ở phòng nào?
  jobTitleId: integer('job_title_id').references(() => jobTitles.id).notNull(), // Chức danh gì? (Trưởng phòng/CV KT-Thuế)
  gradeId: integer('grade_id').references(() => grades.id).notNull(), // Hưởng lương/Cấp bậc nào? (Bậc 4/Bậc 6)
  locationId: integer('location_id').references(() => locations.id).notNull(), // Làm ở đâu? (HCM)
  
  isPrimary: boolean('is_primary').default(true), // Là vị trí chính? (Có thể kiêm nhiệm)
  startDate: timestamp('start_date').notNull(),
  endDate: timestamp('end_date'), // Null = Đang đương nhiệm
  
  status: varchar('status', { length: 50 }).default('ACTIVE'), // ACTIVE, TRANSFERRED, RESIGNED
});

export const userAssignmentsRelations = relations(userAssignments, ({ one }) => ({
  user: one(users, { fields: [userAssignments.userId], references: [users.id] }),
  orgUnit: one(orgUnits, { fields: [userAssignments.orgUnitId], references: [orgUnits.id] }),
  jobTitle: one(jobTitles, { fields: [userAssignments.jobTitleId], references: [jobTitles.id] }),
  grade: one(grades, { fields: [userAssignments.gradeId], references: [grades.id] }),
  location: one(locations, { fields: [userAssignments.locationId], references: [locations.id] }),
}));
```

### 2. Cách dữ liệu của bạn chạy trong Schema này

Hãy lấy ví dụ từ hình của bạn để xem data được Insert vào DB như thế nào:

#### Ví dụ 1: Trần Thị Trúc Đào (Mã NV 001) - Từ Hình 1
*   Bảng `users`: { id: 1, employeeCode: '001', fullName: 'Trần Thị Trúc Đào' }
*   Bảng `orgUnits`: { id: 10, name: 'P.DỊCH VỤ' }
*   Bảng `jobTitles`: { id: 20, name: 'Trợ lý A2' }
*   Bảng `grades`: { id: 8, levelNumber: 8, name: 'Trợ lý A2' } *(Xem hình 3, Trợ lý A2 thuộc Bậc 8)*
*   Bảng `locations`: { id: 1, name: 'HCM' }
*   **Bảng `userAssignments` (Lưu data thực tế):**
    `{ userId: 1, orgUnitId: 10, jobTitleId: 20, gradeId: 8, locationId: 1, isPrimary: true }`

#### Ví dụ 2: Tại sao phải tách `Grade` (Bậc) và `JobTitle` (Chức danh)?
Nhìn vào Hình 3, Bậc 6.
*   **Người A** làm ở P.HCKT.
    *   Grade: Cấp bậc B2 (Bậc 6).
    *   Job Title: "CV.KTTC-HCNS" (Chức vụ cụ thể).
*   **Người B** làm ở P.DỊCH VỤ.
    *   Grade: Cấp bậc B2 (Bậc 6).
    *   Job Title: "Chuyên viên B2" (Gọi tên theo cấp bậc luôn).

Việc tách ra giúp phòng HR tính lương chuẩn (dựa vào Bảng Grades), nhưng phòng ban vẫn có chức danh riêng biệt (dựa vào Bảng JobTitles).


#### Thuật toán vẽ Sơ đồ tổ chức (Image 2)
Trong `drizzle-org-structure.repository.ts`, để lấy toàn bộ sơ đồ công ty (cây phân cấp cha-con), bạn có thể dùng tính năng `with` của Drizzle để đệ quy (recursive query):

```typescript
async getFullOrgChart() {
  const db = this.getDb();
  
  // Lấy gốc (HĐTV - parentId = null) và Include toàn bộ nhánh con
  const chart = await db.query.orgUnits.findMany({
    where: (org, { isNull }) => isNull(org.parentId),
    with: {
      children: {
        with: {
          children: {
            with: {
              children: true // Đệ quy sâu xuống các tầng
            }
          }
        }
      }
    }
  });
  
  return chart;
}
```

### 4. Tích hợp với hệ thống RBAC (Phân quyền) hiện tại của bạn

Trong project của bạn đã có sẵn module `rbac`. Làm sao để kết hợp với Cơ cấu nhân sự này?

**Nguyên tắc: RBAC quyết định "Làm được gì", OrgStructure quyết định "Làm ở dữ liệu nào (Data Scope)".**

Ví dụ: Role `DEPARTMENT_MANAGER` (Trưởng phòng) có quyền `approve:leave_request` (Duyệt nghỉ phép).
Nhưng duyệt nghỉ phép cho ai?
*   Khi API nhận request duyệt phép, hệ thống sẽ:
    1. Check RBAC: User này có role `DEPARTMENT_MANAGER` không? -> Có.
    2. Check OrgStructure: Qua bảng `user_assignments`, lấy được `orgUnitId` của người duyệt (Ví dụ ID=10: P.Dịch Vụ).
    3. Hệ thống chỉ cho phép duyệt đơn của những User có `user_assignments.orgUnitId == 10`.

### TỔNG KẾT
Cấu trúc Database được đề xuất ở trên là **tiêu chuẩn vàng (Golden Standard)** cho các hệ thống ERP / HRM. Nó hoàn toàn linh hoạt, nếu công ty bạn thêm Bậc 11, đổi tên Phòng ban, hoặc thêm Chi nhánh Hà Nội, bạn **chỉ cần cấu hình Data (Thêm Record), không cần sửa bất kỳ dòng code Backend nào**.
