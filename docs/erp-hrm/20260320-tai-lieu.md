# 📘 TÀI LIỆU ONBOARDING & ARCHITECTURE GUIDE
**Dự án:** ERP / HRM SYSTEM
**Phiên bản:** 1.0 | **Stack:** NestJS + TypeScript + Drizzle ORM + PostgreSQL + Redis

---

## 1. TỔNG QUAN DỰ ÁN
Hệ thống ERP/HRM quản lý nhân sự và tài nguyên doanh nghiệp, được thiết kế theo tư tưởng **Clean Architecture** kết hợp **Domain-Driven Design (DDD)**. Hệ thống cho phép dễ dàng mở rộng và sẵn sàng bóc tách thành Microservices khi dự án scale up.

### 1.1 Mục tiêu hệ thống
* **Quản lý nhân sự (HRM):** Tổ chức sơ đồ công ty, vị trí định biên, hồ sơ nhân viên.
* **Phân quyền động (RBAC):** Quản lý Role, Permission theo resource/action, Import/Export cấu hình bằng CSV.
* **Bảo mật đa tầng:** JWT kết hợp Redis Session, OTP, tích hợp Telegram Bot.

### 1.2 Tech Stack

| Thành phần | Công nghệ | Mục đích |
| :--- | :--- | :--- |
| **Framework** | NestJS 10+ | Backend API, DI Container, Module System |
| **Language** | TypeScript 5+ | Type-safety cho toàn bộ codebase |
| **ORM** | Drizzle ORM | Query builder type-safe, Schema-first, không bị phình to như TypeORM |
| **Database** | PostgreSQL 15+ | Cơ sở dữ liệu chính (Primary DB) |
| **Cache & Session**| Redis 7+ | Quản lý JWT Session, OTP, Caching phân quyền RBAC |
| **Auth** | Passport.js + JWT | Chiến lược xác thực, JWT Bearer Token |
| **Chatbot** | Telegraf | Tích hợp Telegram Bot thao tác hệ thống |
| **Logging** | Winston | Ghi log có cấu trúc, Tracking Request-ID, ẩn dữ liệu nhạy cảm |
| **Message Broker** | Custom EventBus | Hỗ trợ Memory (Local), RabbitMQ, Kafka |

## 2. KIẾN TRÚC HỆ THỐNG (CLEAN ARCHITECTURE)
Hệ thống tuân thủ nghiêm ngặt nguyên tắc **Dependency Rule**: Dependency chỉ đi từ ngoài vào trong. Lớp ngoài biết lớp trong, KHÔNG được ngược lại.

### 2.1 Các phân lớp (Layers)
1. **Domain Layer (`domain/`):** Chứa Entities, Value Objects, Domain Events, Interfaces (Ports). Hoàn toàn không phụ thuộc Framework.
2. **Application Layer (`application/`):** Chứa Use Cases (Services), DTOs. Điều phối nghiệp vụ.
3. **Infrastructure Layer (`infrastructure/`):** Chứa Controllers, Drizzle Repositories (Adapters), Mappers. Trực tiếp giao tiếp với Database, HTTP.
4. **Shared Core (`core/shared/`):** Chứa các công cụ dùng chung toàn cục: EventBus, Cache, Transaction Manager, Logger.

### 2.2 Nguyên tắc Port & Adapter
Mọi Dependency quan trọng được abstract qua Interface (Port) và Inject qua NestJS DI.
* VD: `IUserRepository` (Port) $\rightarrow$ `DrizzleUserRepository` (Adapter).
* **Lợi ích:** Dễ dàng thay đổi công nghệ (VD: Đổi gửi Email từ Console sang AWS SES chỉ cần viết 1 Adapter mới mà không cần sửa Core Logic).

## 3. CẤU TRÚC THƯ MỤC
Codebase tổ chức theo **"Feature-first"**. Mỗi module là một đơn vị độc lập.

```text
src/
├── bootstrap/          # App entry (main.ts, app.module.ts)
├── config/             # Config files (Env, Database, Redis, Log...)
├── core/               # Cross-cutting concerns (Filter, Interceptor, Shared Ports/Adapters)
├── database/           # Drizzle setup, schema definitions
│   └── schema/         # DB schema chia theo module (core, rbac, hrm, crm, system)
└── modules/            # Các Business Modules
    ├── auth/           # Đăng nhập, JWT, OTP
    ├── user/           # Quản lý User (Identity)
    ├── rbac/           # Role-based access control
    ├── employee/       # Hồ sơ nhân viên
    ├── org-structure/  # Sơ đồ tổ chức, vị trí định biên
    ├── notification/   # Gửi Email, lắng nghe Domain Events
    └── chatbot-core/   # Telegram bot
```

## 4. DATABASE & THIẾT KẾ DỮ LIỆU (ERD)
Database Schema được khai báo trong `src/database/schema/`. Mỗi module có file schema riêng nhưng được gom chung tại `index.ts` để Drizzle ORM có thể tự động tạo Relational Query.

### 4.1 Các bảng cốt lõi
* **Core:** `users` (Định danh), `sessions` (Phiên làm việc - đồng bộ với Redis).
* **RBAC:** `roles`, `permissions`, `user_roles`, `role_permissions` (Mô hình phân quyền đa tầng).
* **HRM:** `employees` (Hồ sơ nhân viên), `positions` (Vị trí định biên), `job_titles` (Chức danh).

### 4.2 🚀 Kỹ thuật Materialized Path (Sơ đồ tổ chức)
Bảng `org_units` sử dụng kỹ thuật **Materialized Path** (Đường dẫn vật lý) qua trường `path` để lưu cây sơ đồ tổ chức thay vì chỉ dùng đệ quy `parentId`.
* **Cấu trúc:** ID 1 (Công ty) $\rightarrow$ `path: /1/`. ID 3 (Phòng IT thuộc ID 1) $\rightarrow$ `path: /1/3/`.
* **Sức mạnh:** Lấy toàn bộ nhân sự cấp dưới, chi nhánh con/cháu chỉ bằng 1 câu truy vấn siêu tốc: `SELECT * FROM org_units WHERE path LIKE '/1/3/%'`.
* **Thuật toán Move Node:** Khi chuyển phòng ban, dùng SQL `REPLACE(path, oldPath, newPath)` để cập nhật toàn bộ nhánh con ngay lập tức.

## 5. HƯỚNG DẪN TÌM HIỂU MODULE (TỪ DỄ ĐẾN KHÓ)
*Lưu ý cho Developer mới: Để tránh bị ngợp, hãy đọc và hiểu code theo thứ tự cấp độ sau:*

### 🟢 Cấp độ 1: Dễ (Nền tảng & Tiện ích)
* **`NotificationModule`:** Tìm hiểu cách hệ thống chạy bất đồng bộ (Event-Driven). Xem cách `UserRegisteredListener` lắng nghe `UserCreatedEvent` từ EventBus và gọi `IEmailSender` để gửi email.
* **`ChatbotCoreModule`:** Xem cách tích hợp Telegram Telegraf. Cách Bot lưu session vào Redis và nhận diện User bằng `telegramId`.
* **`TestModule`:** Xem Controller Sandbox. Nơi để bạn test thử tắt/mở các quyền (Permissions) một cách dễ dàng thông qua các Endpoints giả lập.

### 🟡 Cấp độ 2: Trung bình (Nghiệp vụ Core)
* **`UserModule`:** Tìm hiểu **Rich Domain Model**. Xem entity `User` có chứa logic như `changePassword()`. Xem cách `UserMapper` gộp dữ liệu từ các bảng rải rác (HRM/CRM) vào chung một thực thể User. Đọc thuật toán Import CSV hàng loạt với Transaction.
* **`OrgStructureModule`:** Nơi chứa thuật toán tạo Cây sơ đồ tổ chức trên RAM (Flatten to Tree) và logic CRUD sử dụng Materialized Path đã nói ở phần 4.
* **`EmployeeModule`:** Hiểu sự tách biệt nguyên lý giữa **Hồ sơ nhân viên (Employee)** và **Tài khoản đăng nhập (User)**. Flow: HR tạo hồ sơ trước $\rightarrow$ IT bấm nút `provision-account` để sinh tài khoản tự động sau.

### 🔴 Cấp độ 3: Khó (Trái tim Bảo mật)
* **`AuthModule`:** Đọc kỹ luồng Login. Hệ thống tạo Token, lưu Session vào **cả PostgreSQL và Redis**. Tìm hiểu cơ chế **Cache Warming** trong `JwtStrategy` (Tìm token trong Redis 1ms, nếu Cache Miss mới xuống Postgres tìm, sau đó lưu ngược lại lên Redis).
* **`RbacModule`:** Hiểu cách hệ thống phân quyền động. Hệ thống tự động lưu mảng quyền của User vào Redis (TTL 300s) để không phải query 5 bảng DB (user_roles $\rightarrow$ roles $\rightarrow$ role_permissions) cho mỗi request. Cơ chế quyền Thượng Phương Bảo Kiếm (keyword `manage:all` hoặc Role `SUPER_ADMIN`).

## 6. QUY TRÌNH & QUY CHUẨN CODE (BEST PRACTICES)

Hãy tuân thủ nghiêm ngặt các quy tắc sau khi tạo tính năng mới:

### 6.1 Tạo một Module mới (Luồng chuẩn)
1. Khai báo Schema trong `database/schema/` -> Chạy lệnh `npx drizzle-kit push`.
2. Tạo thư mục `modules/your-module/domain`. Viết Entity (chứa logic) và Interface Repository Port (`IYourRepo = Symbol()`).
3. Tạo `infrastructure/mappers`. Viết `YourMapper.toDomain()` và `toPersistence()`.
4. Tạo `infrastructure/repositories/drizzle-your.repo.ts`. Implement Interface Port ở bước 2. Bắt buộc dùng `this.getDb(tx)` để hỗ trợ Transaction.
5. Tạo `application/services`. Inject `IYourRepo` và `ITransactionManager`. Bọc logic phức tạp vào `this.txManager.runInTransaction(async (tx) => { ... })`.
6. Tạo `infrastructure/controllers`. Gắn Guard bảo mật.

### 6.2 Cập nhật thêm Phân quyền (Thêm Permission mới)
1. Mở file `src/modules/rbac/domain/constants/rbac.constants.ts` (hoặc các file constants của từng module). Khai báo quyền mới. Vd: `BOOKING_CREATE: 'booking:create'`.
2. Lên Controller, sử dụng Decorator: `@Permissions(PERMISSIONS.BOOKING_CREATE)`.
3. Cập nhật file `rbac.csv` (File gốc cấu hình quyền). Thêm dòng: `STAFF,booking,create,*,Cho phép nhân viên tạo đơn`.
4. Gọi API `POST /api/rbac/data/import` tải file CSV lên để hệ thống tự động nạp quyền vào Database và Redis.

### 6.3 Những ĐIỀU CẤM KỴ (Red Flags 🚩)
* ❌ **KHÔNG** import trực tiếp Adapter (VD: `DrizzleUserRepository`) vào Service. **Bắt buộc** Inject thông qua Symbol Interface (`IUserRepository`).
* ❌ **KHÔNG** gọi Database trực tiếp từ Controller. Mọi Request phải đi qua Service.
* ❌ **KHÔNG** return Entity `User` bị lỗi logic Anemic Model. Hãy đảm bảo Drizzle trả về data, Mapper chuyển thành Class Instance, rồi mới trả về Client (Gọi `.toJSON()`).
* ❌ **KHÔNG** sửa Database Schema mà không báo cho Team hoặc quên chạy Migration.

## 7. HỆ TẦNG DÙNG CHUNG (SHARED CORE)
Nằm tại `core/shared/`, được khai báo Global (`@Global()`). Mọi module có thể Inject trực tiếp.

* **TransactionManager:** Bọc logic vào 1 Transaction của DB.
  ```typescript
  await this.txManager.runInTransaction(async (tx) => {
      await this.userRepo.save(user, tx);
      await this.sessionRepo.create(session, tx);
  });
  ```
* **EventBus:** Truyền tin nội bộ (Memory/RabbitMQ/Kafka). Gọi bằng `await this.eventBus.publish(new UserCreatedEvent(...))`.
* **Redis Cache:** Quản lý bằng `ICacheService` (Set TTL theo giây).
* **Winston Logger:** Masking tự động ẩn thông tin nhạy cảm (password, token), RequestContext tự sinh `requestId` tracking cho mọi dòng log.

## 8. QUY CHUẨN CODE & NHỮNG ĐIỀU "KHÔNG ĐƯỢC LÀM"

Để hệ thống dễ bảo trì, yêu cầu Dev tuân thủ các quy tắc sau:

### ❌ KHÔNG ĐƯỢC:
1. **Tuyệt đối không Import Repository bằng Class Name:** Không dùng `@Inject(DrizzleUserRepository)`, bắt buộc dùng Symbol Interface `@Inject(IUserRepository)`.
2. **Không gọi Database trực tiếp từ Controller:** Luồng bắt buộc: `Controller` $\rightarrow$ `Service` $\rightarrow$ `Repository`.
3. **Không import SharedModule lộn xộn:** `SharedModule` đã là `@Global()`, không cần import lại vào các module con.
4. **Không return Type "any" ở Repository:** Mọi query phải được parse qua `Mapper` (VD: `UserMapper.toDomain()`) trước khi trả về Service.
5. **Không quên Migration:** Nếu sửa Schema DB, bắt buộc phải chạy lệnh Migration trước khi test logic.

### ✅ PHẢI LÀM (Response Standard):
Mọi API (trừ xuất file tải về) phải trả về format chuẩn thông qua `TransformResponseInterceptor`:
```json
{
  "success": true,
  "statusCode": 200,
  "message": "Success",
  "result": { "data": "here" }
}
```

## 9. HƯỚNG DẪN ONBOARDING THỰC HÀNH (SETUP)

### Bước 1: Khởi tạo môi trường
1. Clone repo, chạy `npm install` (khuyên dùng `pnpm install`).
2. Copy `.env.development.example` thành `.env.development`.
3. Khởi động Docker cho Database & Redis:
   ```bash
   docker compose -f infra/docker-compose.dev.yml up -d
   ```

### Bước 2: Chạy Database Migration & Seed
```bash
# Đồng bộ Schema xuống Database
npm run db:push

# Bật cờ RUN_SEEDS=true trong .env.development và chạy App
npm run start:dev
```
*Lưu ý: Seeder (`DatabaseSeeder`) sẽ tự động tạo cấu hình RBAC chuẩn và tài khoản Master.*

### Bước 3: Đăng nhập & Khám phá
* Mở **Swagger UI**: `http://localhost:8080/docs`
* **Tài khoản mặc định:**
  * Username: `superadmin`
  * Password: `123456`
* Đăng nhập ở API `/api/auth/login`, copy `accessToken`, gắn vào nút **Authorize** trên góc phải Swagger để gọi các API khác.

### 💡 Cách thêm một Quyền (Permission) mới:
1. Thêm Hằng số vào `PERMISSIONS` trong `src/modules/rbac/domain/constants/rbac.constants.ts` (VD: `PAYROLL_APPROVE: 'payroll:approve'`).
2. Mở Controller cần bảo vệ, gắn Decorator: `@Permissions(PERMISSIONS.PAYROLL_APPROVE)`.
3. Cập nhật file `rbac.csv` nội bộ (Thêm dòng map Role với Quyền này) $\rightarrow$ Dùng API `POST /api/rbac/data/import` để nạp vào DB.
