## 1. TỔNG QUAN DỰ ÁN (OVERVIEW)

ERP/HRM System là hệ thống quản trị doanh nghiệp và nhân sự lõi, được xây dựng trên nền tảng **NestJS**. Hệ thống không viết theo mô hình MVC truyền thống mà áp dụng kiến trúc **Clean Architecture** kết hợp **Domain-Driven Design (DDD)**.
Mục tiêu của kiến trúc này là giúp dự án có thể dễ dàng mở rộng, bảo trì, và sẵn sàng tách thành các **Microservices** độc lập khi doanh nghiệp phát triển (Scale-up).

### 1.1 Tech Stack (Công nghệ cốt lõi)

| Thành phần | Công nghệ | Mục đích |
| :--- | :--- | :--- |
| **Framework** | NestJS 10+ | Backend API, DI Container, Module System |
| **Language** | TypeScript 5+ | Type-safety cho toàn bộ codebase |
| **ORM** | Drizzle ORM | Query builder type-safe, Schema-first, không bị phình to như TypeORM |
| **Database** | PostgreSQL 15+ | Cơ sở dữ liệu chính (Primary DB) |
| **Cache & Session**| Redis 7+ | Quản lý JWT Session, OTP, Caching phân quyền RBAC |
| **Auth** | Passport.js + JWT | Chiến lược xác thực, JWT Bearer Token |
| **Chatbot** | Telegraf | Tích hợp Telegram Bot thao tác hệ thống |
| **Logging** | Winston | Ghi log có cấu trúc, Tracking Request-ID, ẩn dữ liệu nhạy cảm |
| **Message Broker** | Custom EventBus | Hỗ trợ Memory (Local), RabbitMQ, Kafka |

## 2. KIẾN TRÚC TỔNG THỂ (CLEAN ARCHITECTURE)

Hệ thống áp dụng mô hình **Ports & Adapters (Hexagonal Architecture)**. Nguyên tắc tối thượng: **Dependency (Sự phụ thuộc) chỉ chạy từ ngoài vào trong**. Lớp bên trong không bao giờ được biết đến sự tồn tại của lớp bên ngoài.

### 2.1 Các lớp (Layers) trong mỗi Module

1. **Domain Layer (`domain/`):** Chứa các `Entities` (Thực thể giàu tính năng - Rich Domain Model), `Value Objects`, `Enums` và các Interface `Repositories` (Ports). Lớp này tuyệt đối **KHÔNG** import Drizzle, Express hay bất kỳ thư viện ngoại vi nào.
2. **Application Layer (`application/`):** Chứa `Services` (Use Cases) và `DTOs`. Lớp này điều phối nghiệp vụ. Nó giao tiếp với Database thông qua các Interface (Ports) ở lớp Domain.
3. **Infrastructure Layer (`infrastructure/`):** Lớp ngoài cùng. Chứa `Controllers` (nhận HTTP request), `Mappers`, và các `Repositories` (Adapters) implement các Port từ lớp Domain bằng Drizzle ORM.
4. **Shared Core (`core/shared/`):** Chứa các nền tảng dùng chung toàn cục (Global) như `ITransactionManager`, `EventBus`, `Logger`.

### 2.2 Sức mạnh của Ports & Adapters
Nhờ tiêm phụ thuộc (Dependency Injection) qua các `Symbol` Interface, chúng ta có thể thay đổi công nghệ lõi trong 5 phút mà không sợ hỏng logic:
* Muốn đổi **Gửi Email in ra Console** thành **Gửi qua AWS SES**? -> Chỉ cần viết `SesEmailAdapter` implement `IEmailSender` và đổi chỗ Bind trong Module.
* Muốn đổi **EventBus Local** sang **Kafka**? -> Đổi biến môi trường `EVENT_BUS_TYPE=kafka`.

## 3. CẤU TRÚC THƯ MỤC DỰ ÁN

Codebase được tổ chức theo nguyên tắc **"Feature-first" (Theo tính năng)** thay vì "Type-first" (Theo loại file).

```text
src/
├── bootstrap/          # Entry point: main.ts, app.module.ts
├── config/             # Cấu hình biến môi trường (.env) cho từng module
├── core/               # Cross-cutting concerns (Shared Core)
│   ├── filters/        # Bắt lỗi Global (Exception Filter)
│   ├── interceptors/   # Format Response tự động { success, data, message }
│   └── shared/         # Chứa Ports, EventBus, TransactionManager, Cache
├── database/           # Định nghĩa Schema Drizzle và cấu hình kết nối DB
│   └── schema/         # Schema chia theo module (core, rbac, hrm...)
└── modules/            # CÁC MODULE NGHIỆP VỤ (BUSINESS LOGIC)
    ├── auth/           # Đăng nhập, JWT, Redis Session, Đổi mật khẩu
    ├── rbac/           # Phân quyền động (Roles, Permissions)
    ├── user/           # Quản lý tài khoản (Users)
    ├── org-structure/  # Sơ đồ tổ chức, Vị trí định biên
    ├── employee/       # Hồ sơ nhân sự (HRM)
    ├── notification/   # Xử lý gửi Email/SMS qua Event
    └── chatbot-core/   # Tích hợp Telegram Bot
```

## 4. DATABASE & THIẾT KẾ DỮ LIỆU (ERD)

Database Schema được khai báo trong `src/database/schema/`. Mỗi module có file schema riêng nhưng được gom chung tại `index.ts` để Drizzle ORM có thể tự động tạo Relational Query.

### 4.1 Các bảng cốt lõi
* **Core:** `users` (Định danh), `sessions` (Phiên làm việc - đồng bộ với Redis).
* **RBAC:** `roles`, `permissions`, `user_roles`, `role_permissions` (Mô hình phân quyền đa tầng).
* **HRM:** `employees` (Hồ sơ nhân viên), `positions` (Vị trí định biên), `job_titles` (Chức danh).

### 4.2 🚀 Kỹ thuật Materialized Path (Sơ đồ tổ chức)
Bảng `org_units` sử dụng kỹ thuật **Materialized Path** (Đường dẫn vật lý) qua trường `path` để lưu cây sơ đồ tổ chức thay vì chỉ dùng `parentId`.
* **Cấu trúc:** ID 1 (Công ty) -> `path: /1/`. ID 3 (Phòng IT thuộc ID 1) -> `path: /1/3/`.
* **Sức mạnh:** Lấy toàn bộ nhân sự cấp dưới, chi nhánh con cháu chắt chỉ bằng 1 câu truy vấn siêu tốc: `SELECT * FROM org_units WHERE path LIKE '/1/3/%'`. Không cần đệ quy (Recursive) gây nghẽn DB.

## 5. HƯỚNG DẪN TÌM HIỂU MODULE (TỪ DỄ ĐẾN KHÓ)

Để tránh bị ngợp, Developer mới hãy đọc code theo thứ tự cấp độ sau:

### 🟢 Cấp độ 1: Dễ (Nền tảng & Tiện ích)
* **`NotificationModule`:** Tìm hiểu cách hệ thống chạy bất đồng bộ. Xem cách `UserRegisteredListener` lắng nghe `UserCreatedEvent` từ EventBus và gọi `IEmailSender` để gửi email.
* **`ChatbotCoreModule`:** Xem cách tích hợp Telegram. Cách Bot lưu session vào Redis và nhận diện User.
* **`TestModule`:** Xem Controller Sandbox. Nơi để bạn test thử tắt/mở các quyền (Permissions) một cách dễ dàng.

### 🟡 Cấp độ 2: Trung bình (Nghiệp vụ Core)
* **`UserModule`:** Tìm hiểu **Rich Domain Model**. Xem entity `User` có chứa logic `changePassword()`. Xem cách `UserMapper` gộp dữ liệu từ bảng HRM/CRM vào chung entity User. Xem thuật toán Import CSV hàng loạt.
* **`OrgStructureModule`:** Nơi chứa thuật toán tạo Cây sơ đồ tổ chức trên RAM (Flatten to Tree) và logic Di chuyển phòng ban (Move Node) tự động update trường `path`.
* **`EmployeeModule`:** Hiểu sự tách biệt giữa **Hồ sơ nhân viên (Employee)** và **Tài khoản đăng nhập (User)**. HR tạo hồ sơ, IT bấm nút `provision-account` để sinh tài khoản tự động.

### 🔴 Cấp độ 3: Khó (Trái tim Bảo mật)
* **`AuthModule`:** Đọc kỹ luồng Login. Cách hệ thống tạo Token, lưu Session vào **cả PostgreSQL và Redis**. Tìm hiểu cơ chế **Cache Warming** trong `JwtStrategy` (Tìm token trong Redis 1ms, nếu miss mới xuống Postgres tìm và lưu ngược lại lên Redis).
* **`RbacModule`:** Đọc cách hệ thống phân quyền động. Cách hệ thống tự động lưu mảng quyền của User vào Redis (TTL 300s) để không phải query 5 bảng DB cho mỗi request. Cơ chế quyền Thượng Phương Bảo Kiếm (`manage:all` hoặc `SUPER_ADMIN`).

## 6. QUY TRÌNH & QUY CHUẨN CODE (BEST PRACTICES)

Hãy tuân thủ nghiêm ngặt các quy tắc sau khi tạo tính năng mới:

### 6.1 Tạo một Module mới (Luồng chuẩn)
1. Khai báo Schema trong `database/schema/` -> Chạy lệnh `npx drizzle-kit push`.
2. Tạo thư mục `modules/your-module/domain`. Viết Entity (chứa logic) và Interface Repository Port (`IYourRepo = Symbol()`).
3. Tạo `infrastructure/mappers`. Viết `YourMapper.toDomain()` và `toPersistence()`.
4. Tạo `infrastructure/repositories/drizzle-your.repo.ts`. Implement Interface Port ở bước 2. Bắt buộc dùng `this.getDb(tx)` để hỗ trợ Transaction.
5. Tạo `application/services`. Inject `IYourRepo` và `ITransactionManager`. Bọc logic phức tạp vào `this.txManager.runInTransaction(async (tx) => { ... })`.
6. Tạo `infrastructure/controllers`. Gắn Guard bảo mật.

### 6.2 Cập nhật thêm Phân quyền (Thêm Permission mới)
1. Mở file `src/modules/rbac/domain/constants/rbac.constants.ts` (hoặc các file constants của từng module). Khai báo quyền mới. Vd: `BOOKING_CREATE: 'booking:create'`.
2. Lên Controller, sử dụng Decorator: `@Permissions(PERMISSIONS.BOOKING_CREATE)`.
3. Cập nhật file `rbac.csv` (File gốc cấu hình quyền). Thêm dòng: `STAFF,booking,create,*,Cho phép nhân viên tạo đơn`.
4. Gọi API `POST /api/rbac/data/import` tải file CSV lên để hệ thống tự động nạp quyền vào Database và Redis.

### 6.3 Những ĐIỀU CẤM KỴ (Red Flags 🚩)
* ❌ **KHÔNG** import trực tiếp Adapter (VD: `DrizzleUserRepository`) vào Service. **Bắt buộc** Inject thông qua Symbol Interface (`IUserRepository`).
* ❌ **KHÔNG** gọi Database trực tiếp từ Controller. Mọi Request phải đi qua Service.
* ❌ **KHÔNG** return Entity `User` bị lỗi logic Anemic Model. Hãy đảm bảo Drizzle trả về data, Mapper chuyển thành Class Instance, rồi mới trả về Client (Gọi `.toJSON()`).
* ❌ **KHÔNG** sửa Database Schema mà không báo cho Team hoặc quên chạy Migration.

## 7. SETUP VÀ CHẠY DỰ ÁN LOCAL

### Bước 1: Môi trường
Cài đặt Node.js (v18+), PostgreSQL và Redis. (Khuyên dùng Docker Compose cho DB & Redis).

### Bước 2: Cài đặt và Cấu hình
```bash
# 1. Cài package (Khuyên dùng pnpm)
npm install

# 2. Tạo file môi trường
cp .env.development.example .env.development

# 3. Điền các thông số vào .env.development (DB_URL, REDIS_URI, JWT_SECRET...)
```

### Bước 3: Khởi tạo Dữ liệu (Database Migration & Seeding)
```bash
# Đẩy schema lên Database
npx drizzle-kit push

# Mở file .env.development, set RUN_SEEDS=true
# Sau đó chạy dự án. Hệ thống sẽ tự động tạo tài khoản SUPER_ADMIN.
npm run start:dev
```

### Bước 4: Khám phá API
Mở trình duyệt: `http://localhost:8080/docs` (Swagger UI).
* **Tài khoản mặc định:** Username: `superadmin` | Pass: `123456`.
* Dùng API `/api/auth/login` lấy `accessToken`. Bấm nút **Authorize** trên Swagger để gắn Token vào các request tiếp theo.
